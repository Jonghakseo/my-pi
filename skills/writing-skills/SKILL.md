# writing-skills

## 핵심 원칙

**스킬 작성 = 프로세스 문서의 TDD.**

테스트(압력 시나리오) 작성 → 실패 관찰(베이스라인) → 스킬 작성 → 통과 확인 → 허점 보완.

```
에이전트가 스킬 없이 실패하는 걸 보지 못했다면,
스킬이 올바른 것을 가르치는지 알 수 없다.
```

## Iron Law

```
실패 테스트 없이 스킬을 작성하지 않는다.
```

테스트 전에 스킬을 썼으면 삭제하고 처음부터 다시 한다.
기존 스킬 수정도 동일 — 테스트 없이 수정하면 같은 위반이다.

## 스킬이란

- **스킬**: 검증된 기법, 패턴, 도구의 참조 가이드
- **스킬이 아닌 것**: "이렇게 풀었다"는 이야기 (내러티브)

**생성 기준:**
- 직관적이지 않았던 기법
- 프로젝트 간 재사용 가능
- 다른 사람에게도 유용
- 범용 패턴 (프로젝트 특화 아님)

**생성하지 않을 것:**
- 일회성 해결책
- 이미 잘 문서화된 표준 관행
- 프로젝트 특화 규칙 (→ 메모리에 저장)
- regex/자동화로 강제할 수 있는 것

## TDD 매핑

| TDD 개념 | 스킬 작성 |
|----------|----------|
| 테스트 케이스 | 서브에이전트 압력 시나리오 |
| 프로덕션 코드 | SKILL.md 문서 |
| RED (실패) | 스킬 없이 에이전트가 규칙 위반 |
| GREEN (통과) | 스킬 있으면 에이전트가 규칙 준수 |
| REFACTOR | 새 합리화 발견 → 차단 → 재검증 |

## RED-GREEN-REFACTOR

### RED: 베이스라인 테스트

스킬 **없이** 서브에이전트에게 압력 시나리오를 실행한다:

```
challenger에게:
"[시나리오 설명]. 스킬/가이드 없이 이 상황을 처리해봐."
```

기록할 것:
- 어떤 선택을 했는가?
- 어떤 합리화를 사용했는가? (원문 그대로)
- 어떤 압력에서 위반이 발생했는가?

### GREEN: 최소 스킬 작성

베이스라인에서 발견된 **구체적 합리화/실패만** 해결하는 스킬을 작성한다.
가설적 케이스를 위한 추가 내용은 넣지 않는다.

같은 시나리오를 스킬 **있는** 에이전트로 재실행 → 준수 확인.

### REFACTOR: 허점 메우기

새로운 합리화 발견 시:
1. 합리화 테이블에 추가
2. Red Flags 리스트에 추가
3. 재테스트

## SKILL.md 구조

```markdown
---
name: skill-name-with-hyphens
description: Use when [구체적 트리거 조건]. 워크플로 요약 금지.
argument-hint: "예시 호출 문구"
disable-model-invocation: false
---

# 스킬 이름

## Iron Law / 핵심 원칙
핵심 규칙 1-2문장.

## 언제 사용하나
트리거 조건, 증상, 사용 사례.

## 프로세스 / 핵심 패턴
단계별 절차 또는 패턴 설명.

## Red Flags
위반 징후 목록.

## 합리화 방어
| 핑계 | 현실 | 테이블.

## 주의
경계 조건, 제한 사항.
```

## description 작성 규칙 (CSO)

**Critical:** description은 **트리거 조건만** 적는다. 워크플로를 요약하지 않는다.

```yaml
# ❌ BAD: 워크플로 요약 → 에이전트가 이것만 읽고 스킬 본문을 건너뜀
description: 태스크별 worker 디스패치 후 코드 리뷰하는 스킬

# ✅ GOOD: 트리거 조건만 → 에이전트가 본문을 읽게 됨
description: 구조화된 구현 계획이 있을 때 사용
```

**이유**: 테스트에서 발견됨 — description에 워크플로를 요약하면 에이전트가 본문 대신 description만 따름.

## 스킬 유형별 테스트

### 규율 강제 스킬 (TDD, verification-gate 등)
- 학문적 질문: 규칙을 이해하는가?
- 압력 시나리오: 스트레스 하에서 준수하는가?
- 복합 압력: 시간 + 매몰비용 + 피로 동시 적용
- 합리화 식별 → 명시적 차단 추가

### 기법 스킬 (debugging, pattern 등)
- 적용 시나리오: 기법을 올바르게 적용하는가?
- 변형 시나리오: 엣지 케이스를 처리하는가?
- 정보 부족 테스트: 설명에 빈틈이 있는가?

### 참조 스킬 (API 문서, 명령어 가이드 등)
- 검색 시나리오: 필요한 정보를 찾는가?
- 적용 시나리오: 찾은 정보를 올바르게 사용하는가?
- 빈틈 테스트: 일반적 사용 사례가 커버되는가?

## 합리화 방어

| 핑계 | 현실 |
|------|------|
| "스킬이 명확하잖아" | 작성자에게 명확 ≠ 에이전트에게 명확. 테스트하라. |
| "참조일 뿐이야" | 참조도 빈틈과 불명확 구간이 있다. 테스트하라. |
| "테스트는 과하다" | 미테스트 스킬은 항상 문제가 있다. 15분 테스트가 몇 시간을 절약. |
| "문제 생기면 그때 테스트" | 문제 = 에이전트가 스킬을 사용 못 함. 배포 전 테스트. |
| "시간 없다" | 미테스트 스킬 배포가 나중에 더 많은 시간 소모. |

## 파일 구조

```
skills/
  skill-name/
    SKILL.md              # 메인 참조 (필수)
    supporting-file.*     # 필요 시만
```

- **인라인 유지**: 원칙, 개념, 50줄 미만 코드
- **별도 파일**: 100줄 이상 참조, 재사용 가능 스크립트

## 체크리스트

**RED:**
- [ ] 압력 시나리오 작성 (규율 스킬은 3개 이상)
- [ ] 스킬 없이 실행 — 베이스라인 행동 기록
- [ ] 합리화/실패 패턴 식별

**GREEN:**
- [ ] YAML frontmatter: name + description만 (1024자 이내)
- [ ] description: "Use when..." / "[조건] 시 사용" 형태, 워크플로 요약 금지
- [ ] 베이스라인 실패를 해결하는 최소 내용
- [ ] 스킬 있는 에이전트로 재실행 — 준수 확인

**REFACTOR:**
- [ ] 새 합리화 발견 → 테이블 추가
- [ ] Red Flags 리스트 업데이트
- [ ] 완전할 때까지 재테스트
