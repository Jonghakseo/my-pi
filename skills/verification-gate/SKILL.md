# verification-gate

## Iron Law

```
증거 없이 완료를 주장하지 않는다.
```

"should work", "아마 될 거야", "통과한 것 같다" — 전부 **거짓말**이다.
실행 결과를 직접 보여주지 못하면, 완료가 아니다.

**이 규칙의 문구를 지키는 것이 정신을 지키는 것이다.**

## 게이트 절차

모든 완료 주장 직전에 아래 5단계를 수행한다:

1. **식별**: 이 주장을 증명하는 명령어가 무엇인가?
2. **실행**: 해당 명령어를 **지금** 실행한다 (이전 결과 재사용 금지)
3. **읽기**: 출력 전체를 읽고, exit code를 확인한다
4. **판정**: 출력이 주장을 뒷받침하는가?
   - YES → 증거와 함께 주장
   - NO → 실제 상태를 증거와 함께 보고
5. **그제서야**: 완료를 말한다

어떤 단계든 건너뛰면 = 검증이 아니라 추측이다.

## 주장별 필수 증거

| 주장 | 필요한 증거 | 불충분한 것 |
|------|-----------|-----------|
| 테스트 통과 | 테스트 실행 출력: 0 failures | 이전 실행, "통과할 거야" |
| 빌드 성공 | 빌드 명령 출력: exit 0 | 린터만 통과 |
| 버그 수정됨 | 원래 증상 재현 → 해결 확인 | 코드만 바꿈 |
| 타입체크 통과 | tsc/typecheck 출력: 0 errors | "타입 잘 맞을 거야" |
| 에이전트 완료 | VCS diff로 변경 확인 | 에이전트의 "성공" 보고 |
| 요구사항 충족 | 항목별 체크리스트 대조 | "테스트 통과했으니 됐다" |

## Red Flags — 즉시 멈추고 게이트 수행

다음 중 하나라도 해당되면 **멈추고** 위 절차를 따른다:

- "should", "probably", "아마", "것 같다" 사용
- "Great!", "완벽!", "Done!" 등 만족 표현
- 커밋/푸시/PR 직전
- 서브에이전트 성공 보고를 그대로 신뢰
- 부분 검증으로 전체를 추론
- "이번만", "간단하니까" 같은 합리화

## 합리화 방어

| 핑계 | 현실 |
|------|------|
| "아마 될 거야" | 명령어를 실행하라 |
| "확신한다" | 확신 ≠ 증거 |
| "이번만" | 예외 없음 |
| "린터 통과했으니" | 린터 ≠ 컴파일러 ≠ 테스트 |
| "에이전트가 성공이래" | 독립적으로 검증하라 |
| "피곤하다" | 피곤함 ≠ 면제 |
| "부분 확인으로 충분" | 부분은 아무것도 증명 안 한다 |

## 자동 검증 모드

복잡한 변경의 경우 `verifier` 에이전트를 호출한다:

```
verifier에게: "$ARGUMENTS 의 완료 상태를 검증해줘.
테스트/타입체크/빌드를 실행하고, 주장된 변경이 실제로 존재하는지 확인해줘."
```

verifier 결과가 돌아오기 전에는 완료를 선언하지 않는다.

## 적용 시점

**항상:**
- 완료/성공/통과의 모든 변형
- 만족 표현
- 커밋, PR, 태스크 완료
- 다음 태스크로 이동
- 에이전트 위임 결과 보고

## 최종 규칙

**검증에 지름길은 없다.**

명령어를 실행하고, 출력을 읽고, 그 다음에 결과를 말한다.

비협상 사항이다.
