# /qa — QA 워크플로우

`$ARGUMENTS`(URL 또는 설명)를 대상으로 **브라우저 에이전트 기반 QA**를 수행한다.

---

## Step 0: Prerequisite 확인

### 0-1. browser 에이전트 가용성

`browser` 에이전트와 `agent-browser` CLI가 사용 가능한지 확인한다:

```bash
which agent-browser 2>/dev/null && agent-browser --help 2>&1 | head -1 || echo "NOT_FOUND"
```

**`NOT_FOUND`이면 → STOP:**
> `agent-browser` CLI가 설치되어 있지 않습니다.
> QA 스킬은 브라우저 에이전트가 필요합니다.
> 설치: `npm install -g agent-browser` 또는 프로젝트 문서를 확인하세요.

### 0-2. 세션 이름 생성

QA 전체에서 **하나의 브라우저 세션**을 공유한다:

```bash
QA_SESSION="qa-$(date +%s)"
echo "SESSION: $QA_SESSION"
```

이후 모든 browser 위임에 `--session $QA_SESSION`을 사용한다.

## 모드 선택

시작 시 AskUserQuestion:

> QA 모드를 선택하세요.
>
> A) **Full** — 탐색 + 플로우 테스트 + 이슈 수정 + 재검증 (권장)
> B) **Report Only** — 탐색 + 플로우 테스트 + 이슈 보고만 (코드 수정 안 함)
> C) **Smoke** — 페이지 로드 + 콘솔 에러 + 기본 상태 확인만

---

## Step 1: 대상 파악

### 1-1. URL 확인

`$ARGUMENTS`에서 URL을 추출한다. URL이 없으면 AskUserQuestion:
> QA 대상 URL을 입력해주세요.

### 1-2. 프로젝트 컨텍스트 수집

```bash
git branch --show-current 2>/dev/null
git log --oneline -5 2>/dev/null
```

최근 변경사항을 파악하여 **어디를 집중적으로 테스트할지** 힌트를 얻는다.

### 1-3. 인증 필요 여부

AskUserQuestion:
> 이 사이트에 로그인이 필요한가요?
>
> A) 아니오 — 공개 페이지
> B) 예 — `.env.browser` 크레덴셜 사용
> C) 예 — 직접 로그인 정보 제공

B 선택 시 `~/.pi/agent/agents/.env.browser`에서 크레덴셜을 읽어 `AUTH_INFO`로 보관한다.
C 선택 시 사용자에게 입력 받아 `AUTH_INFO`로 보관한다 (세션 내에서만 사용, 저장 안 함).
A 선택 시 `AUTH_INFO`는 비어 있다.

## Step 2: 페이지 탐색 (Smoke 모드는 여기까지)

`browser` 에이전트에게 위임 (`--session $QA_SESSION` 사용):

**위임 프롬프트 (인증 불필요 시):**
> 세션 `{$QA_SESSION}`에서 {URL}을 열고 다음을 수행해줘:
> 1. 페이지가 정상 로드되는지 확인 (200 상태, 빈 페이지 아닌지)
> 2. `snapshot -i`로 인터랙티브 요소 목록 수집
> 3. `errors`로 콘솔 에러 수집
> 4. `screenshot /tmp/qa-initial.png`으로 초기 상태 캡처
> 5. 페이지의 주요 구조를 텍스트로 요약
>
> 보고 형식: 로드 상태, 인터랙티브 요소 수, 콘솔 에러 수, 페이지 구조 요약

**위임 프롬프트 (인증 필요 시):**
> 세션 `{$QA_SESSION}`에서 먼저 로그인을 수행한 뒤 {URL}을 테스트해줘:
> 1. 로그인 페이지로 이동
> 2. 다음 크레덴셜로 로그인: {AUTH_INFO에서 추출한 로그인 URL, ID, PW}
> 3. 로그인 성공 확인 후 {URL}로 이동
> 4. `snapshot -i`로 인터랙티브 요소 목록 수집
> 5. `errors`로 콘솔 에러 수집
> 6. `screenshot /tmp/qa-initial.png`으로 초기 상태 캡처
> 7. 페이지의 주요 구조를 텍스트로 요약
>
> 보고 형식: 로그인 성공 여부, 로드 상태, 인터랙티브 요소 수, 콘솔 에러 수, 페이지 구조 요약

**Smoke 모드**: 여기서 결과를 보고하고 종료.

## Step 3: 플로우 테스트 (Full / Report Only)

### 3-1. 테스트 시나리오 도출

최근 git 변경사항과 페이지 구조에서 테스트 시나리오를 도출한다:

**자동 도출 기준:**
- 최근 커밋에서 변경된 기능 관련 플로우
- 페이지의 주요 인터랙티브 요소 (폼, 버튼, 네비게이션)
- 인증이 있으면 로그인 플로우

**기본 시나리오 (항상 포함):**
1. 주요 네비게이션 동작 확인
2. 폼이 있으면 빈 제출 → 유효성 검증 확인
3. 폼이 있으면 정상 입력 → 성공 확인 (⚠️ 파괴적 액션 가드 적용)
4. 주요 CTA 버튼 동작 확인 (⚠️ 파괴적 액션 가드 적용)

### ⚠️ 파괴적 액션 가드

다음에 해당하는 동작은 **사용자 명시적 승인 없이 절대 실행하지 않는다:**
- 결제, 구매, 과금이 발생하는 액션
- 데이터 삭제 (계정 삭제, 게시물 삭제 등)
- 메시지/알림 발송 (이메일, SMS, 푸시)
- 외부 서비스 호출 (API 연동, 웹훅 트리거)
- 상태 변경이 되돌릴 수 없는 액션 (주문 취소, 승인 등)

**시나리오에 위 항목이 포함되면:**
1. 해당 단계에 `[⚠️ 파괴적]` 태그를 붙인다
2. 시나리오 확인 시 사용자에게 별도 경고한다
3. 테스트 계정/샌드박스 환경인지 확인한다
4. 승인 없으면 해당 단계를 **건너뛰고** 보고서에 "미테스트 (파괴적 액션)" 표시

**browser 위임 시 항상 포함:**
> 다음 동작은 절대 수행하지 마: 결제 완료, 실제 삭제, 메시지 발송, 되돌릴 수 없는 상태 변경. 확인 다이얼로그가 뜨면 무조건 취소.

시나리오를 사용자에게 제시하고 확인:
> 다음 시나리오를 테스트하겠습니다:
> 1. {시나리오 1}
> 2. {시나리오 2}
> ...
>
> A) 진행 B) 시나리오 수정/추가

### 3-2. 시나리오별 테스트 실행

각 시나리오마다 `browser` 에이전트에게 위임 (`--session $QA_SESSION` 사용):

**세션 상태 확인**: 시나리오 실행 전, 인증이 필요한 사이트라면 로그인 상태를 확인한다. browser 에이전트가 "로그인 필요" 또는 "세션 만료" 신호를 보고하면 Step 2의 인증 프롬프트를 같은 세션에서 재실행한다.

**위임 프롬프트:**
> 세션 `{$QA_SESSION}`에서 다음 시나리오를 테스트해줘:
> 만약 로그인 페이지로 리디렉트되거나 인증 에러가 발생하면, 즉시 보고해줘 (재인증 필요).
>
> **시나리오:** {시나리오 설명}
> **단계:**
> 1. {구체적 단계}
> 2. {구체적 단계}
>
> 각 단계 후:
> - `snapshot -i`로 상태 변화 확인
> - 예상과 다른 동작이 있으면 기록
> - 콘솔 에러가 새로 발생하면 기록
> - 단계 완료 시 `screenshot /tmp/qa-{시나리오}-{단계}.png` 캡처
>
> 보고: 각 단계의 성공/실패, 발견된 이슈, 스크린샷 경로

### 3-3. 반응형 확인

주요 뷰포트에서 레이아웃 확인 (`--session $QA_SESSION` 사용):

**위임 프롬프트:**
> 세션 `{$QA_SESSION}`에서 반응형 레이아웃을 확인해줘:
> 1. `set viewport 375x812` (모바일) → `screenshot /tmp/qa-mobile.png`
> 2. `set viewport 768x1024` (태블릿) → `screenshot /tmp/qa-tablet.png`
> 3. `set viewport 1440x900` (데스크탑) → `screenshot /tmp/qa-desktop.png`
>
> 각 뷰포트에서 깨진 레이아웃, 넘치는 텍스트, 잘린 요소가 있는지 확인.

## Step 4: 이슈 수집 및 분류

모든 테스트 결과를 종합하여 이슈를 분류:

```
QA REPORT — {URL}
══════════════════════════════════
Branch: {브랜치}
Date: {날짜}
Mode: {Full / Report Only / Smoke}

Health Score: {0-100}

ISSUES:
━━━━━━
🔴 Critical (서비스 불가)
  1. {이슈} — {증거: 스크린샷/에러 로그}

🟡 Major (기능 저하)
  1. {이슈} — {증거}

🟢 Minor (개선 사항)
  1. {이슈} — {증거}

SCENARIOS:
━━━━━━━━━
  ✅ {시나리오 1} — 통과
  ❌ {시나리오 2} — 실패 (이슈 #N)
  ✅ {시나리오 3} — 통과

CONSOLE ERRORS: {N개}
  - {에러 메시지 1}
  - {에러 메시지 2}

RESPONSIVE:
  📱 Mobile:  OK / ISSUE
  📋 Tablet:  OK / ISSUE
  🖥️ Desktop: OK / ISSUE

SCREENSHOTS:
  - /tmp/qa-initial.png
  - /tmp/qa-{scenario}-{step}.png
  ...
══════════════════════════════════
```

### Health Score 산정

| 조건 | 감점 |
|------|------|
| Critical 이슈 1개당 | -30 |
| Major 이슈 1개당 | -15 |
| Minor 이슈 1개당 | -5 |
| 콘솔 에러 1개당 | -3 |
| 반응형 깨짐 1건당 | -10 |
| 시나리오 실패 1건당 | -10 |

100점에서 감점. 최소 0점.

**Report Only 모드**: 여기서 보고서를 출력하고 종료.

## Step 5: 이슈 수정 (Full 모드만)

Critical → Major 순서로 수정한다. Minor는 보고만.

### 5-1. 이슈별 수정

각 Critical/Major 이슈에 대해:

1. **원인 분석**: 코드베이스에서 관련 파일 탐색 (`grep`, `read`)
2. **수정**: `worker`에게 위임
3. **테스트 실행**: 기존 테스트가 깨지지 않는지 확인

**위임 프롬프트 (worker):**
> QA에서 발견된 이슈를 수정해줘:
>
> **이슈:** {이슈 설명}
> **증거:** {스크린샷 경로, 콘솔 에러}
> **관련 파일:** {탐색으로 찾은 파일 목록}
>
> 수정 후 테스트를 실행하여 기존 기능이 깨지지 않는지 확인해줘.

### 5-2. 수정 불가 판단

다음 경우 수정을 건너뛴다:
- 외부 서비스 의존 (API 장애, CDN 문제)
- 데이터 문제 (테스트 데이터 부족)
- 대규모 리팩터링 필요 (별도 이슈로 기록)

---

## Step 6: 재검증 (Full 모드만)

수정된 이슈에 대해 `browser` 에이전트로 재검증 (`--session $QA_SESSION` 사용):

**위임 프롬프트:**
> 세션 `{$QA_SESSION}`에서 수정된 이슈를 재검증해줘:
> {수정된 이슈 목록과 재현 단계}
>
> 각 이슈가 해결되었는지 확인하고 스크린샷 캡처.

재검증 실패 시 → 해당 이슈를 "미해결"로 표시.

## 전체 흐름

```
/qa <URL>
  │
  ├─ Step 0: Prerequisite 확인 (agent-browser 가용성, 세션 생성)
  │
  ├─ 모드 선택 (Full / Report Only / Smoke)
  │
  ├─ Step 1: 대상 파악 (URL, 컨텍스트, 인증)
  │
  ├─ Step 2: 페이지 탐색 ← browser 에이전트
  │    └─ Smoke → 여기서 종료
  │
  ├─ Step 3: 플로우 테스트 ← browser 에이전트
  │    ├─ 시나리오 도출 + 승인
  │    ├─ 시나리오별 실행
  │    └─ 반응형 확인
  │
  ├─ Step 4: 이슈 수집 + Health Score
  │    └─ Report Only → 여기서 종료
  │
  ├─ Step 5: 이슈 수정 ← worker 에이전트
  │
  ├─ Step 6: 재검증 ← browser 에이전트
  │
  └─ Step 7: 최종 보고
```

## 주의 사항

- **browser 에이전트의 보고를 그대로 신뢰하지 않는다.** 스크린샷과 구체적 증거를 요구.
- **이슈 수정 범위를 최소화한다.** QA에서 발견된 이슈만 수정. "여기 있는 김에" 리팩터링 금지.
- **외부 서비스 장애를 코드 버그로 오판하지 않는다.** 네트워크/API 에러는 별도 분류.
- **스크린샷은 증거다.** 모든 주요 단계에서 캡처. 이슈 보고 시 반드시 스크린샷 경로 포함.
- **Health Score는 상대적이다.** 절대 기준이 아닌 변화 추적용. 80+ = 양호, 50-79 = 주의, <50 = 위험.
